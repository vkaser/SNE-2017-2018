1 Memory

  1. Read about volatility and its features.
    a) What does Volatility do?
    Volatility is an open source memory forensics framework for incident response and malware analysis. It can be used to analyze memory dumps of 32/64bit OSes (windows, Linux, MacOS and Android) and supports variety of memory file formats for example:

    • Raw Memory using dd
    • Hibernation files
    • Crash Dumps
    • VBox core dump
    • VmWare snapshots
    • QEMU dumps
    • and many other formats.

    In addition, one of the best thing about volatility is the community, as most of the plugins are developed by the community members.

    b) Would Volatility be useful in the acquiring stage?
    No. Since volatility is designed to analyze a memory dump that has been collected by acquisition-based tools.

    c) What parts of Volatility would you use in your investigation on the acquired memory?
    It would depend on what incident happened. But in any case, first it is required to understand what image profile we will use in the future analysis using the imageinfo plugin.

     $ vol.py -f <memory_dump_file> imageinfo

    After that, we might need to acquire information about all processes which were running during the dumping. In this case we are required to use pslist plugin:

     $ vol.py -f <memory_dump_file> --profile=<ImageProfile> pslist

    If we need to get a list of all the processes in the memory, including active, terminated, and hidden processes we should use psscan plugin:

     $ vol.py -f <memory_dump_file> --profile=<ImageProfile> psscan

    In addition, volatility allows to analyse network connection (by using connections, connscan, sockets, and netscan plugins), find malware programs (by using malfind, apihooks, callbacks, mutantscan plugins), know who started the process by using the getsids plugin and the process ID to provide the SID of the user, and many other function.

  2. Look up the used techniques to hide data that the other team used. Use Volatility to explain the memory of the acquired VM. Also think about the PGP encrypted mail.
    In order to analyze memory of the acquired Virtual Machine (VM) with Volatility, we need to do a memory dump of running VM. With the option dumpvmcore --filename <name> of VBoxManage, we can create a system dump of the running VM, which will be written into the given file. This file will have the standard ELF core format (with some custom sections).
    First, lets do the memory dump into the ELF file:

    $ vboxmanage debugvm "Windows_7" dumpvmcore --filename dump.elf
$ ls -alh dump.elf 
 -rw------- 1 sergey sergey 2,1G апр 23 15:01 dump.elf

    We’re interested into the first LOAD section, that’s where main memory reference is. We can get the correct offset using objdump:

     $ objdump -h dump.elf | egrep -w "(Idx|load1)"
 Idx Name          Size      VMA               LMA               File off  Algn
   1 load1         80000000  0000000000000000  0000000000000000  00002598  2**0

    So memory dump is in dump.elf, starting at offset 0x2598 and counting 0x80000000 bytes (2048Mb)

    Now let’s extract the RAM, getting rid of the first bytes:

     $ size=0x80000000;off=0x2598;head -c $(($size+$off)) dump.elf | tail -c +$(($off+1)) > dump.raw

    Now the file dump.raw contains a memory image that can be analyzed with Volatility:

     $ volatility -f dump.raw imageinfo
